# flash-promoter 手机 App PRD

> 文档类型：产品需求文档 / PRD  
> 项目名称：flash-promoter  
> 模块名称：手机 App / Mobile Companion App  
> 版本：v1.0  
> 目标：定义 flash-promoter 手机端的产品定位、核心流程、功能范围、架构边界与验收标准  

---

## 1. 文档目的

`flash-promoter` 的桌面端定位是多平台内容适配与发布工作台，负责完整内容编辑、平台版本生成、校验、模拟发布、发布日志和后续真实平台接入。

手机 App 不应简单复制桌面端能力，也不应做成复杂后台系统。手机端应作为 `flash-promoter` 的移动伴随端，承担轻量内容输入、移动端预览、平台版本确认、复制分享、辅助发布和发布记录查看等任务。

本 PRD 旨在明确手机 App 的产品边界：

```text
桌面端：完整内容生产与平台适配工作台
手机端：轻量输入、移动预览、版本确认、复制分享、辅助发布
```

---

## 2. 产品定位

### 2.1 一句话定位

`flash-promoter Mobile` 是 flash-promoter 的手机端伴随应用，帮助创作者在手机上快速记录内容、预览各平台版本、确认发布素材，并通过复制、分享面板或平台 App 跳转完成辅助发布。

### 2.2 核心判断

手机端适合做：

- 快速写草稿
- 粘贴内容
- 拍照 / 选图
- 查看平台版本
- 修改标题、摘要、标签、话题
- 确认各平台版本
- 复制平台内容
- 调起分享面板
- 打开平台 App
- 查看发布记录
- 作为团队审核确认入口

手机端不适合做：

- 复杂公众号长文排版
- 大规模多账号管理
- 复杂平台凭证配置
- B站大视频上传与分片任务
- 桌面浏览器自动化
- 绕过平台登录、验证码或风控
- 自动点击其他 App 的最终发布按钮

## 3. 产品目标

### 3.1 用户目标

用户可以在手机上完成以下任务：

1. 快速输入或粘贴内容。
2. 选择要适配的平台。
3. 生成或查看各平台版本。
4. 对各平台版本进行轻量编辑。
5. 确认内容是否可发布。
6. 复制平台内容或调用系统分享能力。
7. 查看发布记录和模拟发布结果。
8. 在移动端完成最终人工确认。

### 3.2 产品目标

手机端应实现以下目标：

1. 降低创作者移动端记录和整理内容的成本。
2. 让用户不打开电脑也能查看、确认和轻编辑平台版本。
3. 为小红书、知乎等移动使用场景提供更自然的辅助发布路径。
4. 与桌面端形成分工，而不是重复做复杂功能。
5. 保留未来与桌面端、Web 端、服务端同步的扩展能力。

## 4. 非目标与边界

### 4.1 MVP 阶段不做

手机 App MVP 不做以下功能：

1. 不做完整桌面端功能复制。
2. 不做复杂双栏编辑器。
3. 不做真实全平台一键发布。
4. 不做自动控制其他平台 App 发布。
5. 不绕过登录。
6. 不绕过验证码。
7. 不绕过平台风控。
8. 不管理公众号 AppSecret、B站开放平台密钥等复杂凭证。
9. 不做团队权限系统。
10. 不做复杂数据分析。
11. 不做大视频分片上传。
12. 不做浏览器自动化脚本。

### 4.2 合规边界

手机端必须遵守：

1. 不使用非公开隐藏接口作为核心能力。
2. 不自动点击平台 App 的最终发布按钮。
3. 不绕过平台规则。
4. 不伪装用户行为进行自动化批量发布。
5. 对于复制和分享，必须明确提示用户自行检查后发布。
6. 发布结果中必须区分“已复制 / 已分享 / 已辅助打开”和“真实发布成功”。
7. 如果没有真实平台回执，不得声称“已发布成功”。

## 5. 用户画像

### 5.1 个人创作者

特点：

- 经常在手机上产生灵感。
- 会在微信、备忘录、社交平台中临时写内容。
- 需要快速整理成多平台版本。

核心需求：

- 快速新建草稿。
- 粘贴内容后自动生成平台版本。
- 手机上快速改标题、标签和小红书文案。
- 一键复制到平台 App。

### 5.2 小红书 / 知乎创作者

特点：

- 主要使用手机发布。
- 关注标题、封面文案、标签、话题。
- 内容通常较短，适合移动端编辑。

核心需求：

- 自动生成小红书标题、正文、话题。
- 自动生成知乎文章或回答版本。
- 支持复制文案和打开平台 App。
- 支持图文卡片素材管理。

### 5.3 视频创作者

特点：

- 视频内容可能在桌面端制作，但标题、简介、标签可以在手机端确认。
- 发布前需要快速检查 B站标题、简介、标签。

核心需求：

- 查看 B站稿件信息。
- 修改标题、简介和标签。
- 查看发布状态。
- 作为发布前确认端。

### 5.4 团队审核者

特点：

- 不一定参与内容创作。
- 主要在手机上查看内容并给出确认意见。

核心需求：

- 查看平台版本。
- 检查标题、摘要、标签。
- 标记通过或退回修改。
- 查看发布记录。

团队审核在 MVP 阶段可以只做“本地确认状态”，不做完整权限系统。

## 6. 核心使用场景

### 6.1 场景一：手机快速新建内容

用户在路上想到一个选题，打开手机 App：

```text
打开 App
→ 点击新建内容
→ 输入标题和正文
→ 添加标签
→ 保存草稿
```

后续可以在手机端继续生成平台版本，也可以同步到桌面端继续深度编辑。

### 6.2 场景二：生成小红书版本并辅助发布

用户输入一段内容，希望发小红书：

```text
输入内容
→ 选择小红书
→ 生成小红书版本
→ 查看标题、正文、话题、封面文案
→ 修改确认
→ 复制文案 / 调起分享面板 / 打开小红书
→ 用户手动发布
```

系统只负责准备内容，不替用户绕过平台发布流程。

### 6.3 场景三：手机端查看桌面端生成的版本

用户在桌面端生成了四个平台版本，然后在手机端检查：

```text
打开手机 App
→ 查看待确认内容
→ 进入平台版本
→ 检查公众号 / 知乎 / B站 / 小红书版本
→ 标记确认
```

### 6.4 场景四：复制知乎版本

用户想把内容发到知乎：

```text
打开知乎版本
→ 查看标题和正文
→ 点击复制标题
→ 点击复制正文
→ 打开知乎 App 或网页
→ 用户粘贴并手动发布
```

### 6.5 场景五：查看发布记录

用户想确认某篇内容是否已经处理过：

```text
打开发布记录
→ 查看内容标题
→ 查看各平台状态
→ 查看模拟发布结果
→ 查看复制 / 分享 / 辅助打开记录
```

## 7. 产品形态

### 7.1 推荐形态

手机端建议采用以下产品形态：

```text
移动端 Companion App
```

即：手机端不是完整替代桌面端，而是桌面端的移动补充。

### 7.2 端的分工

| 能力 | 桌面端 | 手机端 |
|---|---|---|
| 长文编辑 | 强 | 弱 |
| 快速记录 | 中 | 强 |
| 平台版本生成 | 强 | 中 |
| 平台版本预览 | 强 | 强 |
| 轻量编辑 | 中 | 强 |
| 复杂排版 | 强 | 弱 |
| 公众号草稿配置 | 强 | 弱 |
| B站大视频上传 | 强 | 弱 |
| 小红书辅助发布 | 中 | 强 |
| 知乎辅助发布 | 中 | 强 |
| 发布日志查看 | 强 | 强 |
| 团队确认 | 中 | 强 |

---

## 8. 手机端主流程

手机端主流程应保持线性，但比桌面端更轻。

```text
新建 / 选择内容
→ 输入或编辑内容
→ 选择平台
→ 生成平台版本
→ 逐个平台确认
→ 复制 / 分享 / 辅助打开
→ 查看结果
```

## 10. 页面需求

## 10.1 首页

### 页面目标

让用户快速开始一篇内容，或继续处理已有内容。

### 页面内容

- App 名称：flash-promoter
- 主按钮：新建内容
- 最近草稿
- 待确认内容
- 最近发布记录
- 快速入口：小红书笔记、知乎回答、B站简介、公众号图文

### 交互要求

1. 点击“新建内容”进入内容输入页。
2. 点击最近草稿进入编辑。
3. 点击待确认内容进入平台版本确认页。
4. 首页不要展示复杂技术状态。

---

## 10.2 新建内容页

### 页面目标

让用户快速输入内容。

### 字段

- 标题
- 摘要，可选
- 正文
- 标签，可选
- 图片，可选

### 输入模式

支持：

- 普通文本
- Markdown
- 富文本粘贴

默认：

```text
普通文本
```

### 交互要求

1. 用户可以直接输入。
2. 用户可以从剪贴板粘贴。
3. 用户可以从相册选择图片。
4. 用户可以拍照添加图片。
5. 用户可以保存草稿。
6. 用户可以进入下一步选择平台。

### 主按钮

```text
下一步：选择平台
```

### 校验

- 标题不能为空。
- 正文不能为空。
- 正文过短时给出提示，但可保存草稿。

---

## 10.3 选择平台页

### 页面目标

让用户选择需要生成版本的平台。

### 平台列表

- 微信公众号
- 知乎
- B站
- 小红书

### 卡片内容

每个平台卡片展示：

- 平台名称
- 简短说明
- 默认动作
- 开关

### 默认动作文案

| 平台 | 展示文案 | 内部模式 |
|---|---|---|
| 微信公众号 | 生成图文草稿素材 | draft |
| 知乎 | 生成文章 / 回答文案 | assist |
| B站 | 生成标题、简介和标签 | simulate |
| 小红书 | 生成笔记和话题 | assist |

### 交互要求

1. 默认全选。
2. 用户可以取消平台。
3. 至少选择一个平台。
4. 不展示 adapter、simulate、draft、assist 等技术词。

### 主按钮

```text
生成平台版本
```

## 10.4 生成平台版本页

### 页面目标

生成选中平台的适配内容。

### 页面状态

#### 生成中

显示：

- 生成进度
- 当前平台
- 简短说明

#### 生成完成

显示平台版本卡片。

每个平台卡片包含：

- 平台名称
- 生成状态
- 标题预览
- 标签 / 话题数量
- 查看按钮

### 交互要求

1. 生成完成后显示概览。
2. 用户可以点入单个平台查看详情。
3. 支持重新生成单个平台。
4. 不要一次性展开所有平台全文。

## 10.6 确认与校验页

### 页面目标

集中展示各平台是否已确认，以及是否可以进入辅助发布。

### 页面内容

每个平台一张卡片：

- 平台名称
- 确认状态
- 校验状态
- 错误
- 警告
- 推荐操作

### 校验状态

- 通过
- 有警告
- 有错误

### 交互规则

1. 有错误的平台不能进入辅助发布。
2. 有警告的平台可以继续，但需要提醒。
3. 未确认的平台需要先确认。
4. 所有操作必须写入日志。

## 10.7 复制 / 分享 / 辅助打开页

### 页面目标

帮助用户将内容转移到平台 App 或其他发布入口。

### 支持操作

#### 复制

- 复制标题
- 复制正文
- 复制摘要
- 复制标签
- 复制全部内容

#### 分享

- 调用系统分享面板
- 分享文字
- 分享图片，后续
- 分享图文素材包，后续

#### 打开平台

- 打开知乎
- 打开小红书
- 打开 B站
- 打开微信

### 重要提示

每次辅助操作后需要提示：

```text
已为你准备内容。请在平台内检查后手动发布。
```

如果只是打开平台或复制内容，状态应记录为：

```text
已辅助打开
已复制
已分享
```

不得记录为：

```text
已发布成功
```

除非未来接入真实官方发布回执。

## 12. 技术架构建议

### 12.1 推荐技术路线

如果已有桌面端基于 TypeScript / React，手机端建议优先考虑：

```text
React Native + Expo
```

原因：

- TypeScript 复用成本较低。
- 适合快速开发移动端。
- 支持相册、分享、剪贴板、深链跳转。
- 可以逐步扩展为原生 App。
- 不需要一开始处理过多原生工程复杂度。

也可以考虑：

```text
Capacitor + React
```

适合希望最大化复用 Web 前端的情况。

### 12.2 推荐项目结构

建议未来项目结构演进为：

```text
flash-promoter/
  apps/
    desktop/
    mobile/
    web/

  packages/
    core/
    adapters/
    ai-adapter/
    renderer/
    asset-pipeline/
    storage/
    shared-ui/
```

手机端应尽量复用：

- `packages/core`
- `packages/ai-adapter`
- `packages/renderer`
- 通用类型定义
- 平台校验逻辑

手机端不应依赖：

- Electron
- Playwright 桌面自动化
- 桌面端 SQLite 直接访问实现
- 桌面文件系统 API

### 12.3 本地存储

MVP 阶段可以使用手机本地存储：

- SQLite
- AsyncStorage
- MMKV

建议：

```text
结构化数据：SQLite
轻量设置：MMKV / AsyncStorage
图片文件：App 沙盒文件系统
```

### 12.4 同步策略

MVP 阶段可以不做云同步。

后续同步路径：

```text
手机端
→ API Server
→ 数据库
→ 桌面端 / Web 端
```

或者：

```text
桌面端本地服务
→ 局域网同步
→ 手机端
```

但局域网同步对普通用户配置成本较高，不建议作为第一优先级。

## 15. UI 设计原则

### 15.1 移动优先

手机端不要照搬桌面端布局。

必须避免：

- 双栏编辑器
- 大量表格
- 多面板同时展示
- 复杂 Dashboard
- 开发调试信息暴露

### 15.2 单屏单任务

每个页面只做一件事。

例如：

- 输入页只负责输入。
- 平台选择页只负责选择。
- 平台详情页只负责查看和编辑单个平台。
- 发布记录页只负责查看记录。

### 15.3 卡片式平台管理

平台版本应以卡片形式展示：

```text
微信公众号卡片
知乎卡片
B站卡片
小红书卡片
```

用户点击卡片进入详情。

### 15.4 操作文案用户化

不要在手机端暴露技术词。

不推荐：

- simulate
- draft
- assist
- adapter
- mock
- publish job

推荐：

- 模拟发布
- 生成草稿素材
- 辅助发布
- 复制内容
- 打开平台
- 已记录

## 25. 总结

`flash-promoter` 手机 App 的核心价值不是替代桌面端，而是补足移动场景。

它应该承担：

```text
随手记录
→ 快速生成
→ 移动预览
→ 轻量确认
→ 复制分享
→ 辅助打开平台
```

它不应该承担：

```text
复杂排版
→ 平台凭证管理
→ 全自动真实发布
→ 浏览器自动化
→ 绕过平台机制
```

手机端的正确方向是：

```text
轻量、线性、卡片化、移动优先、用户最终确认
```

这样既能提升创作者移动端发布效率，又不会让产品陷入复杂、不可控或不合规的自动化发布风险。
